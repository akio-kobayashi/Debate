# ローカルLLMディベートデモ 基本仕様

## 1. 目的

ローカルLLMに次の3役を担当させ、ブラウザ上でディベートの進行を分かりやすく表示する。

- A：テーマに賛成する討論者
- B：テーマに反対する討論者
- C：中立的なファシリテーター

初版では、LLMの高度な自律性よりも、発言者・発言順・現在の状態が観客に明確に伝わることを優先する。

テーマは任意の分野から入力できる。ただし、入力文字列をそのままA・Bへ渡すのではなく、最初のCのターンで議論可能な命題、対象範囲、前提、評価観点へ整理する。役割プロンプトは固定し、テーマに応じてこの整理結果をターンごとのコンテキストへ差し込む。

## 2. 初版で決定する構成

```text
ブラウザ（クライアント画面）
        │ HTTPS / Tailscale Serve または NetBird
        ▼
Debate API・Web画面サーバー
        │ localhost
        ▼
Ollamaサーバー（LLM推論）
```

### 2.1 サーバー

- Ollamaを動かす推論用マシンを1台用意する。
- 同じマシン上でDebate APIを動かす。
- Debate APIだけをTailscaleまたはNetBird経由でクライアントに公開する。
- Ollama APIは原則として `127.0.0.1:11434` で待ち受け、外部クライアントには直接公開しない。
- Debate APIからOllamaへはローカル接続する。
- A・B・Cは同じモデルを使用し、役割ごとにシステムプロンプトを切り替える。
- 同時生成は行わず、プログラムで発言順を固定して逐次実行する。

### 2.2 オーバーレイネットワーク（Tailscale / NetBird）

- TailscaleまたはNetBirdのどちらか一方を、サーバーとクライアントの接続手段として選択する。
- インターネットへポートフォワーディングしない。
- TailscaleではACLまたはGrantで、利用を許可したクライアントだけがDebate APIへ接続できるようにする。
- NetBirdではAccess Policyで、許可したクライアントからサーバーのTCP 8000番だけを許可する。
- Tailscaleの推奨構成は、`tailscale serve` でlocalhostのDebate APIをHTTPS公開する方式とする。
- 直接接続方式では、8000番のホスト側ファイアウォールもオーバーレイ側に限定する。
- セットアップは `server/scripts/setup_ubuntu.sh --overlay tailscale` または `--overlay netbird` で切り替える。

### 2.3 クライアント

- クライアントはWebブラウザとする。
- ブラウザはDebate APIが提供する画面を表示する。
- ブラウザからOllamaへ直接アクセスしない。
- 発言生成中は、LLMから届いた文字列をストリーミング表示する。

## 3. 初版の利用フロー

1. 利用者がブラウザでDebate画面を開く。
2. テーマを入力する、または用意されたテーマを選択する。
3. 「開始」を押す。
4. 最初の「次の発言」でCがテーマを定義・正規化し、整理後の議題と評価観点を表示する。
5. 「次の発言」を押すたびに、固定された順序で1回だけLLMを呼び出す。生成中は次の操作を無効にする。
6. 生成中の発言を該当するA・B・Cパネルへ表示する。
7. 全ターン終了後、Cの最終整理を表示して終了する。
8. 「リセット」で新しいテーマのディベートを開始できる。

テーマが軽微に曖昧な場合は、Cが仮定を明示して進行する。議論可能な命題へ整理できない場合は、A・Bへ進まず、利用者にテーマの修正を求める。

初版は手動進行を基本とする。自動進行は後から追加する。

## 4. ディベート進行

### 4.1 固定ターン

初版のターンは次の9ターンとする。

| ターン | 話者 | 内容 |
|---:|:---:|:---|
| 1 | C | 生テーマの正規化、用語の定義、議論の条件・評価観点提示 |
| 2 | A | 賛成側の立論 |
| 3 | B | 反対側の立論 |
| 4 | C | 両者の主張整理、反論すべき論点の提示 |
| 5 | A | Bの立論への反論 |
| 6 | B | Aの立論への反論 |
| 7 | A | 最終弁論 |
| 8 | B | 最終弁論 |
| 9 | C | 論点、合意点、未解決点の最終整理 |

UI上は「ラウンド」として表示してもよいが、初版の内部状態は9ターンで管理する。

### 4.2 発言制約

- 1回の発言は初版では最大300日本語文字程度を目安とする。
- 1回の発言で新しい論点を増やしすぎない。
- AとBには、それぞれの立場の根拠と相手への応答を求める。
- Bには代替案または条件を求める。
- Cは賛否を表明せず、与えられた発言の整理だけを行う。
- 発言順、ターン数、終了条件はLLMではなくDebate APIが管理する。

### 4.3 テーマコンテキスト

Turn 1完了後、Debate APIは次のThemeContextをセッションに保存する。

```text
raw_theme
motion
definitions
scope
assumptions
evaluation_axes
clarification_needed
```

Turn 2以降のプロンプトには、`raw_theme`ではなくThemeContextを主な議論対象として渡す。ThemeContextはLLMが作成する整理結果であり、事実確認済みの情報ではない。Turn 1後は、A・Bが都合よく議題を変更できないよう固定する。

## 5. 画面仕様

```text
┌──────────────────────────────────────────────┐
│ ローカルLLMディベート       接続: 正常  2/9   │
│ テーマ：                                      │
│ [                                        ]    │
│ [開始] [次の発言] [停止] [リセット]           │
├────────────────┬────────────────┬────────────┤
│ A 賛成側       │ B 反対側       │ C 進行役   │
│ 青系           │ 赤系           │ 中立色     │
│                │                │            │
│ 最新の発言     │ 最新の発言     │ 論点整理   │
│                │                │ 次の指示   │
├────────────────┴────────────────┴────────────┤
│ 発言履歴                                      │
│ C → A → B → C ...                             │
├──────────────────────────────────────────────┤
│ 状態：Aが生成中 / 待機中 / 完了 / エラー      │
└──────────────────────────────────────────────┘
```

### 5.1 必須表示

- 現在のテーマ
- 現在のラウンドまたはターン番号
- A・B・Cの固定された3パネル
- 各パネルの役割と立場
- 最新発言のストリーミング表示
- 発言履歴
- 現在の状態（待機中、生成中、完了、エラー）
- 開始、次の発言、停止、リセットの操作
- 「LLMによる生成結果であり、事実確認済みではない」旨の注意書き

### 5.2 初版では表示しないもの

- 動的な論点マップ
- 自動投票
- 勝者の断定
- 客観性を保証した点数評価
- 複数観客による同時参加
- トークン単位の詳細な技術メトリクス
- 複数モデルの比較

## 6. 状態管理

ディベートは次の状態を持つ。

```text
idle → ready → generating → waiting → generating → ... → finished
                       └──→ stopping → waiting
                       └──→ clarifying → ready
                       └──────────────→ error
```

- `idle`：テーマ未入力または初期状態
- `ready`：テーマが入力され、開始可能
- `generating`：現在の話者の発言を生成中
- `stopping`：生成停止要求を受け、Ollamaの処理終了を待っている
- `waiting`：発言が完了し、次の発言を待っている
- `clarifying`：Cが議論可能なテーマへ整理できず、利用者の修正を待つ
- `finished`：最終整理まで完了
- `error`：Ollamaまたは通信でエラーが発生

`generating` 中は、同じセッションに対する「次の発言」を受け付けない。`stopping` 中も次の発言を受け付けず、`turn_stopped` イベントを受信してから再開する。

`clarifying` 中はA・Bの生成を受け付けない。テーマ修正後はThemeContextと発言履歴を初期化して`ready`へ戻す。

### 6.1 LLM同時実行数

- サーバー全体でLLM生成は常に1件だけ実行する。
- 待ち行列は作らない。別セッションが実行中の場合は `429 Too Many Requests` を返す。
- クライアントは `turn_completed` または `turn_stopped` を受信するまで「次の発言」を無効にする。
- クライアントの待機制御だけに依存せず、サーバー側でも必ず実行中状態を検査する。

## 7. APIの最小仕様

初版はREST APIとストリーミングレスポンスで構成する。通常のAPI payloadはJSON、発言生成中のイベントはSSEとし、各イベントのデータ部分はJSONとする。

### `POST /api/debates`

新しいディベートを作成する。

```json
{
  "theme": "大学の授業で生成AIの使用を認めるべきである"
}
```

レスポンスには `debate_id`、テーマ、状態、使用モデルを含める。

### `POST /api/debates/{debate_id}/next`

次のターンの生成を開始する。生成結果はこのレスポンスでは返さず、`GET /api/debates/{debate_id}/events` のSSEで配信する。

正常時は `202 Accepted` を返し、ターン番号、話者、状態を含める。

- 同じセッションが生成中または停止処理中の場合は `409 Conflict`
- サーバー全体で別のLLM生成が実行中の場合は `429 Too Many Requests`
- レスポンスに `Retry-After` を付けてもよい

### `GET /api/debates/{debate_id}/events`

ディベートセッションのイベントをSSEで購読する。クライアントはセッション作成後、最初の「次の発言」より前に接続する。

少なくとも次のイベントを送る。

- `turn_started`
- `token`
- `turn_completed`
- `turn_stopped`
- `theme_context_ready`
- `theme_clarification_required`
- `debate_finished`
- `error`

SSEイベントには `id` を付ける。接続が切れた場合は、再接続時に現在状態を `GET /api/debates/{debate_id}` で取得する。イベント再送が必要になった場合は `Last-Event-ID` 対応を追加する。

### `GET /api/debates/{debate_id}`

現在の生テーマ、ThemeContext、状態、ターン番号、発言履歴を取得する。

### `POST /api/debates/{debate_id}/theme`

`clarifying`中、またはTurn 1をやり直したい場合にテーマを修正する。

```json
{
  "theme": "修正後のテーマ"
}
```

テーマ、ThemeContext、発言履歴、ターン番号を初期化して`ready`へ戻す。A・B・Cの生成中は受け付けない。

### `POST /api/debates/{debate_id}/stop`

生成中の発言を中止する。停止要求は生成処理のロックを待たずに受け付け、`202 Accepted` を返す。

- 状態を `stopping` にする
- Ollamaのストリームへキャンセルを要求する
- 完全な終了処理はバックグラウンドで行う
- 終了後に `turn_stopped` をSSEで送る

### `POST /api/debates/{debate_id}/reset`

セッションを初期状態へ戻す。

## 8. Ollama接続仕様

Debate APIの設定値として、少なくとも次を持つ。

```text
OLLAMA_BASE_URL=http://127.0.0.1:11434
OLLAMA_MODEL=gemma4:31b
```

- モデル名は初版ではサーバー側の設定で固定する。
- ブラウザからモデル名を自由に指定できないようにする。
- Ollamaが停止している場合は、画面に接続エラーを表示する。
- モデルの応答がJSONである必要はない。通常の発言テキストをストリーミングする。
- Cの最終整理など、構造化が必要になった機能は後の段階でJSON出力を検討する。

## 9. 保存方針

初版では永続データベースを使用しない。

- 1セッションの状態はDebate APIのメモリ上で管理する。
- 必要に応じて、終了時にJSONまたはMarkdownで保存できる設計にする。
- サーバー再起動後のセッション復元は初版の対象外とする。

## 10. エラー時の表示

次の場合は、LLMの発言として扱わず、画面にエラー状態を表示する。

- Ollamaサーバーに接続できない
- 指定モデルが存在しない
- 生成がタイムアウトした
- 生成中に停止した
- 同じセッションで操作が競合した

エラーには利用者向けの短い説明を表示し、詳細はサーバーログに残す。

## 11. 初版の受け入れ条件

- TailscaleまたはNetBird内の許可されたブラウザから画面を開ける。
- テーマを入力してディベートを開始できる。
- 「次の発言」でA・B・Cが仕様どおりの順に発言する。
- 生成中のテキストが該当パネルへ逐次表示される。
- 発言履歴に話者とターンが残る。
- 生成中の二重実行を防止できる。
- Ollama停止時に、画面が固まらずエラーを表示する。
- リセット後に別のテーマで再実行できる。

## 12. 初版の対象外・保留事項

- 使用モデル（初期予定は `gemma4:31b`。実機での速度・メモリ確認後に最終確定）
- OllamaサーバーのOS、GPU、量子化方式
- Tailscale ACL/GrantまたはNetBird Access Policyの利用者・端末一覧
- 認証方式（Tailnet参加だけで十分か、アプリ認証を追加するか）
- 自動進行
- 聴衆からの質問入力
- 論点の構造化表示
- 複数セッションの同時実行
- 議論の採点や勝敗判定
- 履歴の永続保存

これらは、最小構成が動作した後に追加仕様として決める。

## 13. クライアント基本仕様

初版ブラウザ画面の具体的なデザインは [BROWSER_UI_DESIGN.md](BROWSER_UI_DESIGN.md) に定義する。

- クライアント端末は選択したオーバーレイネットワークへ接続したノートPCとする。
- 初版の表示はブラウザを第一候補とする。
- ブラウザはTailscale ServeのHTTPS URL、またはTailscale/NetBirdのプライベートIPでDebate APIへ接続する。
- クライアントはOllamaの `11434` 番ポートへ直接接続しない。
- SSEは `GET /api/debates/{debate_id}/events` で購読する。
- 「次の発言」は `POST` で開始し、結果はSSEで受け取る。
- クライアントは `turn_completed` または `turn_stopped` を受け取るまで次の発言を待つ。
- 専用アプリを追加する場合も、ブラウザと同じREST API・SSEを使用する。
- ターン順、LLM同時実行数、セッション状態はサーバーを正とする。

## 14. プロンプト基本仕様

A・B・Cの役割プロンプト、9ターンごとの指示、履歴の渡し方、出力検証は [PROMPT_SPEC.md](PROMPT_SPEC.md) に定義する。

- A・Bは本文のみを生成する。
- Cはラベル付きの整理結果を生成する。
- Cの出力から次の話者やターンを決定しない。
- A・B・Cの固定system promptと、テーマ依存のThemeContextを分離する。
- Turn 1のCでThemeContextを作り、Turn 2以降のuser promptへ差し込む。
- LLMの内部思考過程は画面に表示しない。
